Per D-19: In `supabase/functions/sync/index.ts`, find the per-Switcher processing loop where `filterEvents()` is called on raw calendar events. AFTER the `filterEvents()` call, add a second filter step that removes declined meetings.

The Google Calendar API returns `attendees` as an array of `{ email: string, responseStatus: string, ... }`. The `responseStatus` values are: `"needsAction"`, `"declined"`, `"tentative"`, `"accepted"`.

Add the declined filter:
```typescript
// Filter declined meetings (D-19)
const afterDeclinedFilter = filteredEvents.filter(event => {
  if (!event.attendees || event.attendees.length === 0) return true;
  const switcherAttendee = event.attendees.find(
    (a: { email?: string; responseStatus?: string }) => a.email === switcher.email
  );
  // If switcher explicitly declined, exclude the event
  if (switcherAttendee && switcherAttendee.responseStatus === 'declined') {
    return false;
  }
  return true;
});
```

Use `afterDeclinedFilter` (or whatever the existing variable naming convention is — match the style) as the input to the next pipeline stage (title parsing / classification). The `switcher.email` field should already be available from the Switcher record fetched from Supabase. Check the `types.ts` Switcher interface for the email field name.

If the `attendees` field is not currently included in the Google Calendar API fetch, check `calendarFetcher.ts` — the field is returned by default when using `fields=items(...)`. No changes to calendarFetcher.ts should be needed since attendees is a standard field.
