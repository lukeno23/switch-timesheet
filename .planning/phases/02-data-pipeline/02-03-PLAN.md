**Step 1: Create `supabase/functions/_shared/googleAuth.ts`** (per D-30, PIPE-01)

Implement the manual JWT signing pattern from 02-RESEARCH.md Pattern 1. This avoids the google-auth-library npm compatibility issue (Research Pitfall 1).

```typescript
export async function getGoogleAccessToken(
  serviceAccountJson: string,
  userEmail: string
): Promise<string>
```

Implementation (per Research Pattern 1):
1. Parse the service account JSON string
2. Build JWT header: `{ alg: "RS256", typ: "JWT" }`
3. Build JWT claims: `{ iss: sa.client_email, scope: "https://www.googleapis.com/auth/calendar.readonly", aud: "https://oauth2.googleapis.com/token", exp: now + 3600, iat: now, sub: userEmail }`
4. Base64url-encode header and payload (use standard `btoa` available in Deno)
5. Import the private key using `crypto.subtle.importKey("pkcs8", ...)` with algorithm `{ name: "RSASSA-PKCS1-v1_5", hash: "SHA-256" }`
6. Sign the `header.payload` string using `crypto.subtle.sign("RSASSA-PKCS1-v1_5", cryptoKey, data)`
7. Base64url-encode the signature
8. Construct the JWT: `header.payload.signature`
9. Exchange JWT for access token via POST to `https://oauth2.googleapis.com/token` with `grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer` and `assertion=jwt`
10. Return `access_token` from response

IMPORTANT: Use base64url encoding (replace `+` with `-`, `/` with `_`, remove `=` padding) for JWT components, not standard base64. This is required by the JWT spec.

The private key PEM must be stripped of `-----BEGIN PRIVATE KEY-----` / `-----END PRIVATE KEY-----` headers and newlines before base64-decoding to get the raw PKCS#8 DER bytes.

**Step 1: Create `supabase/functions/_shared/outputValidator.ts`** (per D-10, CLAS-03)

```typescript
export interface ValidationResult {
  valid: boolean;
  client_name: string;
  category: string;
  department: string;
  issues: string[];
}

export function validateClassifications(
  results: Array<{ row: number; client: string; category: string; department: string }>,
  validClients: Set<string>,
  validCategories: Set<string>,
  validDepartments: Set<string>
): Array<ValidationResult>
```

For each result:
- Check `client` is in `validClients` set (case-insensitive comparison). If not, mark invalid with issue "unknown client: {value}".
- Check `category` is in `validCategories` set (case-insensitive). If not, mark invalid with issue "unknown category: {value}".
- Check `department` is in `validDepartments` set (case-insensitive). If not, mark invalid with issue "unknown department: {value}".
- Return the original values for valid results, the issues list for invalid ones.

Valid departments are: "Marketing", "Design", "Brand", "PM", "Management", "Cross-Department".

export async function classifyWithLLM(
  events: ParsedEvent[],

  validCategories: string[],

  validDepartments: string[],

): Promise<Array<{ row: number; client: string; category: string; department: string }>>

Implementation:
1. Batch events (batch size = 80 per Claude's discretion, matching classify_with_ai.py approach)
2. For each batch, call Gemini 2.5 Flash REST API at `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent`
3. Use structured output (Research Pattern 4): `responseMimeType: "application/json"` + `responseSchema` with `enum` fields for category and department
4. Set `temperature: 0.1` for deterministic classification
5. API key from parameter (loaded from Deno.env.get in the sync orchestrator, per D-30)

System prompt (port from classify_with_ai.py, enhanced with full context per D-06):
- Include the COMPLETE category list with descriptions from Legend.pdf / instructions.md
- Include the COMPLETE client list with aliases
- Include Switcher department mappings so LLM knows context
- Include classification rules/heuristics from instructions.md
- Instruct: "Classify each event by assigning client, category, and department. Use the canonical names exactly as provided."

User message format (for each event in the batch):
```
Row {index}: Switcher: {name} ({dept}), Title: "{title}", Task: "{taskDetails}", Client field: "{clientRaw}"
```

Add a 6-second delay between batches to respect the 10 RPM Gemini free tier limit.

6. Parse the structured JSON response (guaranteed valid by responseSchema)
7. Validate results using `validateClassifications` from outputValidator.ts
8. Invalid results get classification_method = "misc" (rejected, flagged for review)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Edge Function -> Google Calendar API | Service account JWT impersonation; credentials are server-side secrets |
| Edge Function -> Gemini API | API key for LLM calls; server-side secret |
| Google Calendar -> Rule engine | Untrusted event data enters classification pipeline |
| Gemini -> DB | LLM output must be validated before DB write |
