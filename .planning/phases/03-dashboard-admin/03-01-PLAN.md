**File 1: `supabase/functions/_shared/authVerifier.ts`**
Shared auth helper that verifies the `x-switch-auth` header against the `SWITCH_AUTH_HASH` Edge Function secret. Returns null if authorized, or a 401 Response with CORS headers if not. Pattern follows sync/index.ts `authenticateRequest` but checks a hash header instead of Bearer token.

```typescript
export const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers': 'authorization, x-client-info, apikey, content-type, x-switch-auth',
  'Access-Control-Allow-Methods': 'POST, OPTIONS',
};

export function verifyAuthHash(req: Request): Response | null {
  const authHash = req.headers.get('x-switch-auth');
  const expectedHash = Deno.env.get('SWITCH_AUTH_HASH');
  if (!authHash || !expectedHash || authHash !== expectedHash) {
    return new Response(JSON.stringify({ error: 'Unauthorized' }), {
      status: 401,
      headers: { ...corsHeaders, 'Content-Type': 'application/json' },
    });
  }
  return null;
}
```

Note: Define corsHeaders here rather than importing from `@supabase/supabase-js/cors` to ensure `x-switch-auth` is in the allowed headers list. Export corsHeaders so `admin/index.ts` can use them.

**File 3: `supabase/functions/admin/index.ts`**
Multiplexed admin Edge Function. Structure:

1. `Deno.serve(async (req) => { ... })` entry point
2. OPTIONS preflight returns 200 with corsHeaders
3. Call `verifyAuthHash(req)` -- return error response if auth fails
4. Parse body as `{ action, ...payload }`
5. Switch on action string to route to handler functions
6. Each handler does: validate input, check uniqueness/FK constraints, execute supabaseAdmin query, return JSON response

All 11 actions per RESEARCH.md action table:
- `create-switcher`: INSERT into switchers (name, email, primary_dept, is_management_member). Validate email uniqueness (D-11: "A Switcher with that email already exists.").
- `update-switcher`: UPDATE switchers by id. Same email uniqueness check excluding self. If `active=false`, check upcoming events count and return warning with `requireConfirmation: true` if count > 0 (D-11 deactivation warning).
- `create-client`: INSERT into clients (name, target_hourly_rate). Validate name uniqueness.
- `update-client`: UPDATE clients by id (name, target_hourly_rate, active).
- `create-alias`: INSERT into client_aliases (client_id, alias). Validate alias uniqueness with specific error: "That alias is already assigned to {Client Name}." (D-11).
- `delete-alias`: DELETE from client_aliases by id.
- `create-category`: INSERT into categories (name, department, llm_hint). Validate name uniqueness and department value against allowed list: ['Brand', 'Design', 'Digital', 'Film', 'Management', 'Production'].
- `update-category`: UPDATE categories by id (name, department, llm_hint, active).
- `create-billing`: INSERT into client_billing. Compute `eur_equivalent` server-side: if currency='EUR' then amount, if currency='USD' then amount * fx_rate_to_eur. Validate client_id FK exists.
- `update-billing`: UPDATE client_billing by id. Same EUR equivalent computation.
- `delete-billing`: DELETE from client_billing by id (hard delete per D-18).
- `trigger-sync`: Route sync trigger through admin function to keep SYNC_SECRET server-side. Internally fetch the sync Edge Function URL with the SYNC_SECRET header.

Helper function `jsonResponse(body, status)` wraps `new Response(JSON.stringify(body), { status, headers: { ...corsHeaders, 'Content-Type': 'application/json' } })`.

Every error response uses plain-English messages per D-11. No raw Postgres error strings -- catch Postgres errors and translate to user-friendly messages.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Admin Edge Function | Untrusted client sends CRUD mutations; auth hash is the only gate |
| Admin Edge Function -> Supabase | Service role key bypasses RLS; function must validate all inputs |
| Browser -> Supabase (anon reads) | RLS SELECT policies are the boundary; no write access for anon |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | admin/index.ts | mitigate | Verify x-switch-auth header matches SWITCH_AUTH_HASH env var before any action |
| T-03-02 | Tampering | admin/index.ts | mitigate | Server-side validation of all inputs: email uniqueness, alias uniqueness, FK existence, enum constraints, computed eur_equivalent |
| T-03-03 | Information Disclosure | admin/index.ts | mitigate | Return plain-English errors only; catch Postgres errors and translate; never expose service role key or raw query details |
| T-03-04 | Elevation of Privilege | client_billing RLS | mitigate | Anon SELECT only policy; no INSERT/UPDATE/DELETE for anon or authenticated roles; writes only via service role in Edge Function |
| T-03-05 | Denial of Service | admin/index.ts | accept | Low risk: 6 management users, shared password; no public API surface |
| T-03-06 | Information Disclosure | authVerifier.ts | mitigate | Timing-safe comparison not used (hash comparison via ===) but acceptable: hash is SHA-256, no partial information leaked, and attacker must already know the password to generate a valid hash |
</threat_model>
